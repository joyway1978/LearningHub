## 问题陈述

公司内部缺少一个集中的AI学习资料分享平台。之前的网站功能简陋，无法统计员工对学习内容的 engagement（点赞、浏览），也难以发现优质内容。

## 目标用户

- **内容贡献者**：愿意分享AI学习资料的员工
- **内容消费者**：希望系统学习AI知识的员工

## 现状

现有网站支持基础文件上传，但：
- 无法记录浏览数
- 缺少点赞互动功能
- 没有统计面板
- 员工不知道哪些内容受欢迎

## 最简可行产品（MVP）

**核心功能**：
1. 视频上传与在线播放
2. PDF 上传与在线预览（PPT 转换延期到第二期）
3. 浏览数统计
4. 点赞功能
5. 员工登录认证

**技术栈**：
- 前端：Next.js 14 + Tailwind CSS
- 后端：Python FastAPI
- 数据库：MySQL
- 文件存储：MinIO
- 部署：Docker Compose

## 约束条件

- 内部系统，无需复杂权限管理
- 所有员工可上传/查看
- 不需要审批流程
- 自托管，数据不外流

## 前提假设

1. 平台的核心价值是发现和传播优质AI学习资料 — 员工能通过浏览数和点赞数快速找到有用的内容，上传者能获得认可。✅
2. 需要登录认证，但不需要复杂的权限管理，所有员工都可以上传和查看，不需要审批流程或部门隔离。✅
3. 第一期先做视频和PPT，文档/图片可后续迭代。✅

## 考虑的方案

### 方案A：Supabase快速方案
- **特点**：托管服务，零运维
- **优点**：1-2天上线
- **缺点**：依赖第三方
- **决策**：放弃，要求自托管

### 方案B：全栈自托管分离架构（选定）
- **特点**：Next.js + FastAPI + MySQL + MinIO
- **优点**：完全自主可控，数据在公司内部
- **缺点**：需要运维资源
- **决策**：采纳

### 方案C：飞书/钉钉小程序
- **特点**：复用IM基础设施
- **优点**：无需额外登录
- **缺点**：依赖特定平台
- **决策**：放弃，需要独立网站

## 推荐方案

采用**方案B：全栈自托管分离架构**

**上传流程**（经过后端中转）：
1. 前端选择文件，上传到 FastAPI 后端
2. 后端进行文件校验（类型、大小、安全扫描）
3. 后端将文件流式传输到 MinIO
4. 后端写入数据库，返回成功

**文件大小限制**：
- 视频：最大 500MB
- PDF：最大 50MB

**支持格式**：
- 视频：mp4, webm
- 文档：pdf

## API设计

### 认证
- POST /auth/login - 登录
- POST /auth/refresh - 刷新Token

### 课件
- GET /materials - 列表（分页、排序：created_at, view_count, like_count）
- GET /materials/:id - 详情
- POST /materials - 创建课件记录（不处理文件）
- DELETE /materials/:id - 删除（软删除，状态改为 hidden）
- POST /materials/:id/like - 点赞/取消点赞（toggle 机制）

### 上传
- POST /upload - 上传文件（multipart/form-data）
  - 请求：file, title, description
  - 响应：material_id, file_url
  - 后端处理：校验 → 存MinIO → 写数据库

## 关键设计决策

### 浏览数统计规则
- 同一用户在 10 分钟内重复访问同一课件，只计 1 次浏览
- 未登录用户按 IP 地址去重（10分钟窗口）
- 浏览数实时更新，使用数据库计数器

### 认证方案
- 第一期使用 JWT + 自建简单账号体系
- 登录态 7 天有效，支持自动刷新
- 后续可对接公司 LDAP/SSO

### PDF 预览
- 浏览器使用原生 PDF 渲染预览
- 第一期直接上传 PDF 文件，不提供 PPT 转换
- 缩略图异步生成

### 视频播放
- 第一期直接播放原视频（支持 MP4/WebM 格式）
- 不做转码，降低复杂度
- 后端支持 HTTP Range 请求，实现视频分段加载
- 第二期考虑 HLS 多清晰度适配

### 缩略图生成（异步）
- 视频：使用 ffmpeg 截取第一帧
- PDF：使用 pdf2image 或类似工具提取第一页
- 采用后台异步任务处理，避免上传接口超时
- 第一期使用 Python ThreadPoolExecutor 实现简单队列
- 第二期升级为 Celery + Redis 队列

### 异步任务队列（第一期临时方案）
```python
# 使用 ThreadPoolExecutor 实现简单后台任务
from concurrent.futures import ThreadPoolExecutor
executor = ThreadPoolExecutor(max_workers=3)

# 提交缩略图生成任务
executor.submit(generate_thumbnail, material_id, file_path)
```

### 定时清理任务
- 每天运行一次，清理超过 30 分钟仍处于 'processing' 状态的记录
- 删除未关联数据库记录的 MinIO 孤儿文件
- 清理临时上传目录中的过期文件

### MySQL 连接池配置
```python
# SQLAlchemy 连接池配置（针对大文件上传优化）
pool_size=20           # 基础连接数
max_overflow=30        # 额外连接
pool_timeout=30        # 获取连接超时（秒）
pool_recycle=3600      # 连接回收时间（秒）
```

### 统一错误响应格式
```json
{
  "error": {
    "code": "FILE_TOO_LARGE",
    "message": "文件大小超过限制（最大500MB）",
    "details": { "max_size": 524288000, "actual_size": 629145600 }
  }
}
```

**错误码定义**：
- `FILE_TOO_LARGE` - 文件超过大小限制
- `UNSUPPORTED_FORMAT` - 不支持的文件格式
- `UPLOAD_FAILED` - 上传过程中失败
- `INVALID_TOKEN` - Token 无效或过期
- `UNAUTHORIZED` - 未授权访问
- `NOT_FOUND` - 资源不存在
- `RATE_LIMITED` - 请求过于频繁
- `INTERNAL_ERROR` - 服务器内部错误

## 审查者关注的问题

对抗性审查评分：**7/10**

### 需要重点关注的风险

**后端中转上传的风险**：
- **带宽瓶颈**：后端服务器承担双向流量，需确保足够带宽
- **内存溢出**：500MB视频必须用流式处理+磁盘临时文件，禁止内存缓冲
- **超时风险**：大文件可能超过HTTP超时，需增加超时配置
- **并发控制**：需限制同时上传数和单用户上传速率
- **存储双写**：临时文件+MinIO双重存储，需及时清理临时文件

### 建议补充的实现细节

**文件上传**：
- 使用磁盘临时文件，避免内存缓冲
- 支持上传进度反馈（SSE 或分片上传）
- 添加文件类型深度校验（magic number）
- 集成 ClamAV 病毒扫描（可选）

**PDF 支持**：
- 第一期只支持 PDF 上传，不提供 PPT 转换（简化部署）
- 用户自行将 PPT 另存为 PDF 后上传
- 第二期考虑添加 PPT 转 PDF 功能（使用 LibreOffice）

**事务一致性**：
```
采用"数据库记录优先"策略：
1. 先写数据库，状态为 'processing'
2. 流式上传到 MinIO
3. 更新数据库状态为 'active'
4. 定时任务清理 'processing' 超过30分钟的记录及其临时文件
```

**视频播放优化**：
- 后端支持 HTTP Range 请求，实现视频分段加载
- 浏览器使用原生 video 标签，支持缓冲和预加载

### 部署注意事项

- ffmpeg 需要包含在 Docker 镜像中（用于视频缩略图）
- MinIO bucket 需要配置 CORS 策略
- MySQL 连接池需要足够大（考虑上传长连接）
- 临时文件目录需要定期清理和监控磁盘空间

### 已确认的设计决策

| 问题 | 决策 | 理由 |
|------|------|------|
| 视频大小限制 | 保持 500MB，后端中转 | 安全检查优先于带宽优化 |
| PPT 支持 | 第一期只支持 PDF | 避免 LibreOffice 重型依赖，简化部署 |
| 事务一致性 | 定时清理孤儿文件 | 简单可靠，每天运行一次 |
| 缩略图生成 | 异步后台任务 | 避免上传接口超时，提升用户体验 |
| 异步队列 | 第一期 ThreadPoolExecutor(max_workers=3)，第二期 Celery+Redis | 渐进式演进 |
| 连接池 | pool_size=20, max_overflow=30 | 支持大文件上传的长连接 |
| 错误响应 | 统一 JSON 格式，包含 code/message/details | 前后端统一处理 |
| 测试策略 | 完整测试金字塔（单元+集成+E2E） | 不可妥协的测试覆盖 |
| 视频播放 | FastAPI 代理 Range 请求到 MinIO | 统一认证层 |
| 浏览统计 | 直接写入数据库 | 内部系统 QPS 不高，避免过度设计 |
| 缩略图并发 | ThreadPoolExecutor(max_workers=2) | 限制资源使用，防止过载 |

### 关键实现要求

1. **流式上传**：使用磁盘临时文件，禁止内存缓冲
2. **分片上传**：大文件使用 MinIO multipart 分片上传
3. **异步缩略图**：ThreadPoolExecutor(max_workers=3)
4. **定时清理**：每天清理 'processing' 超过 30 分钟的记录
5. **数据库索引**：已包含高频查询索引
